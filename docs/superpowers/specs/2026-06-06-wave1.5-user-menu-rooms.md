## 3. Billing Room (`/billing`)

### Problem

`Billing.tsx` uses `Frame` as its shell. `Frame` renders the full topbar (HLogo wordmark + entry counter + theme toggle + security dot + user menu). This is the correct shell for the Tapestry view вЂ” it is not appropriate for a focused account subpage where the user came from the user menu.

### Fix

Replace `Frame` with `ClothShell`. Topbar:

- Left: `в†ђ heirloom` вЂ” a `<Link to="/loom">` in monospace uppercase, bone-faint
- Center: `billing`

All billing content and functionality is preserved unchanged вЂ” only the shell wrapper changes.

### Constraints

- `ClothShell` does not render the entry counter or user menu вЂ” this is correct for an account subpage
- Remove the `Frame` import; add `ClothShell` import from `../loom/components/ClothShell`
- Remove any `Frame`-specific props (`left=`, `right=`, `showEdge=`)

## 4. Family Room (`/family`)

### Problem

`Family.tsx` uses `ClothShell` (correct) and the `DYE_VARS` / `DYE_TEXT` system (correct). However, the `FamilyMember` interface has an `avatarUrl` field, and the page may render avatar circles вЂ” a В§2.6 anti-pattern.

### Fix

Audit `Family.tsx` for avatar circle rendering. Replace any `avatarUrl`-based circle (typically a `border-radius: 50%` or `rounded-full` element) with:

- **3px solid dye left-border** on the member row вЂ” use `member.dye ? DYE_VARS[member.dye] : DYE_VARS['iron']`
- **Member name in DYE_TEXT color** вЂ” `member.dye ? DYE_TEXT[member.dye] : 'var(--bone-dim)'`
- The `avatarUrl` field can remain on the interface (it comes from the API) but must not be rendered

### Member row structure after fix

```
[3px dye border] [name in DYE_TEXT color]  В·  [relationship]      [edit]
                 [email if present]
                 [role/status if present]
```

## 5. Settings Audit (`/settings`)

### Problem

`Settings.tsx` uses `ClothShell` and is generally well-structured. A targeted pass is needed to catch any design constitution violations introduced during the settings feature build.

### What to check and fix

1. **Border-radius** вЂ” all `borderRadius` values must be 0px. Input fields max 2px. Never 4px+.
2. **Inline hex colors** вЂ” any raw hex (`#b07a4a`, `#f4ecd8`, etc.) must be replaced with CSS tokens (`var(--warm)`, `var(--bone)`, etc.). Exception: values not in the token set (rare) may stay.
3. **Spinners** вЂ” any spinner/loader must be replaced with a 1px hairline progress bar (`background: var(--warm), height: 1px, position: fixed top`).
4. **Toasts** вЂ” any toast/snackbar must be replaced with inline status text.
5. **`savedFlash` feedback** вЂ” check the existing `savedFlash` state renders inline (e.g., a monospace "saved" line below the button), not as a toast.

If no violations are found, no changes are made and the task is recorded as "no changes needed."
